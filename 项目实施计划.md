# 考研英语长难句分析与检测项目实施计划

## 1. 项目定位

本项目定位为：

> 考研英语长难句结构训练系统

它不是普通翻译工具，也不是单纯背单词工具。核心目标是让考生看到：

- 一个长难句的主干在哪里；
- 定语、状语、从句、插入语等修饰成分如何嵌套；
- 英文结构块和中文结构直译之间如何对应；
- 重点词和重点短语在本文语境中的意思；
- 考研阅读中这类句子可能如何被检测。

第一版先做 MVP，但代码结构要为后续完整产品预留扩展空间。

MVP 的意思是 Minimum Viable Product，即“最小可行产品”。在本项目中，它不是随便做个演示，而是先做出一版真正能用、能验证学习体验的核心系统。

## 3. 总体设计原则

### 3.1 MVP 不等于写死

第一版可以只用本地 JSON，不做数据库、不做登录、不做后台，但代码结构不能写成一次性页面。

必须遵守：

- 页面组件不直接依赖某个具体 JSON 文件；
- 所有数据读取都通过 repository 或 service；
- 所有数据结构都有 TypeScript 类型；
- JSON 进入页面前要通过 schema 校验；
- 练习题生成逻辑独立于页面；
- 高亮逻辑独立于数据读取；
- 后续接数据库时，尽量只替换数据层，不重写页面。

### 3.2 当前数据先适配，不强行改造

当前 JSON 是句子数组。MVP 阶段不要急着修改数据文件本身，而是在代码里做一层 adapter，将它包装成内部统一的 `Article` 模型。

当前文件：

```text
Sentence[]
```

项目内部使用：

```text
Article
  id
  title
  year
  examType
  textId
  sentences
```

这样后续多篇文章、数据库、后台管理都能自然接上。

### 3.3 解析数据和练习数据分离

句子解析数据负责：

- 原句；
- 结构直译；
- 结构块；
- 重点词；
- 重点短语；
- 修饰关系。

练习题数据负责：

- 题干；
- 选项；
- 答案；
- 解析；
- 题型；
- 用户作答记录。

第一版可以从解析数据自动生成基础练习题，但后续复杂练习系统必须独立建模。

### 3.4 AI 功能后置

AI 自动批改很有价值，但不应该放在第一阶段。

原因：

- 当前最重要的是先验证长难句结构训练体验；
- AI 批改需要稳定的评分标准；
- AI 批改成本高，且需要保存用户答案和评分记录；
- 没有用户系统时，AI 批改结果无法形成长期学习闭环。

所以 AI 批改应在用户登录、练习系统、数据库之后再接入。

### 6.2 Sentence 模型

```ts
export interface Sentence {
  sentenceId: string;
  year: number;
  examType: string;
  textId: string;
  original: string;
  difficulty: number;
  translationLiteral: string;
  chunks: Chunk[];
  keyWords: KeyWord[];
  keyPhrases: KeyPhrase[];
}
```

### 6.3 Chunk 模型

```ts
export interface Chunk {
  chunkId: string;
  role: string;
  english: string;
  chinese: string;
  parentId: string | null;
  targetId: string | null;
  relation: string | null;
}
```

字段含义：

- `chunkId`：结构块唯一编号；
- `role`：语法角色，例如主语、谓语、宾语、定语从句；
- `english`：英文结构块；
- `chinese`：该结构块的中文直译；
- `parentId`：当前结构块属于哪个更大的结构块；
- `targetId`：当前结构块修饰、说明或依附哪个结构块；
- `relation`：面向学生的自然语言解释。

### 6.4 KeyWord 模型

```ts
export interface KeyWord {
  word: string;
  meaning: string;
  chunkId: string;
}
```

### 6.5 KeyPhrase 模型

```ts
export interface KeyPhrase {
  phrase: string;
  meaning: string;
  chunkId: string;
}
```

## 7. 数据校验规则

第一版必须做 JSON 校验，避免后续数据越多越难排错。

建议用 `zod` 建立 schema。

### 7.1 句子级校验

必须校验：

- `sentence_id` 必须存在；
- `original` 必须存在；
- `difficulty` 必须是数字；
- `translation_literal` 必须存在；
- `chunks` 必须是数组；
- `key_words` 必须是数组；
- `key_phrases` 必须是数组。

### 7.2 chunk 级校验

必须校验：

- `chunk_id` 必须唯一；
- `role` 必须存在；
- `english` 必须存在；
- `chinese` 必须存在；
- `parent_id` 如果不为 null，必须能在当前句子的 chunks 中找到；
- `target_id` 如果不为 null，必须能在当前句子的 chunks 中找到。

### 7.3 重点词和重点短语校验

必须校验：

- `word` 或 `phrase` 必须存在；
- `meaning` 必须存在；
- `chunk_id` 必须能在当前句子的 chunks 中找到；
- `word` 或 `phrase` 最好能在 `original` 中找到，找不到时给出 warning，而不是直接报错。

### 7.4 高亮匹配校验

每个 chunk 的 `english` 最好能在原句中匹配到。

但是不能强制全部成功，因为有些 chunk 可能是：

- 带引号的表达；
- 省略了一部分内容；
- 原句里有换行、连字符、多个空格；
- chunk 是抽象结构，不一定完全连续。

因此策略是：

- 完全匹配成功：用于高亮；
- 规范化空格后匹配成功：用于高亮；
- 匹配失败：仍然展示结构表，但不给英文原句高亮，并在开发环境 warning。

## 10. 高亮系统设计

### 10.1 为什么不能完全依赖字符串匹配

当前 chunk 中有：

```json
"english": "King Juan Carlos of Spain"
```

可以在原句中直接找到。

但后续会遇到：

- 原文多个空格；
- 换行；
- 连字符；
- 引号不同；
- 大小写差异；
- chunk 不是连续文本；
- 同一个短语在句子中重复出现。

所以高亮逻辑要独立封装。

### 10.2 第一版匹配策略

函数：

```ts
matchChunkPositions(sentence.original, sentence.chunks)
```

返回：

```ts
type ChunkPosition = {
  chunkId: string;
  start: number;
  end: number;
  confidence: "exact" | "normalized" | "failed";
};
```

匹配顺序：

1. 原文直接匹配；
2. 统一空格后匹配；
3. 统一引号后匹配；
4. 忽略大小写匹配；
5. 失败则返回 failed。

第一版只需要支持高亮能匹配上的 chunk。匹配失败的 chunk 仍然在结构表里展示。

### 10.3 后续升级方案

后续如果数据量变大，可以在导入数据库时自动生成：

```text
chunk_start
chunk_end
```

这些字段由程序生成，不建议让大模型生成，因为大模型容易数错字符位置。

## 15. AI 自动批改扩展设计

AI 自动批改建议在数据库、用户系统、练习系统之后接入。

### 15.1 AI 批改场景

- 学生的整句翻译；

- 学生对主干的判断；

### 15.2 AI 批改输入


- 是否漏译关键成分。

## 16. 每个单词完整标注扩展设计

当前数据只标重点词和重点短语。以后如果要实现“点击每一个单词都有解释”，需要增加 token 层。

### 16.1 Token 模型

```ts
export interface Token {
  tokenId: string;
  sentenceId: string;
  text: string;
  lemma: string;
  pos: string;
  meaningInContext: string;
  baseMeaning?: string;
  difficulty?: number;
  start: number;
  end: number;
  chunkId?: string;
  phraseId?: string;
}
```

### 16.2 Token 标注策略

不要一开始手写全部 token。建议分三步：

1. 程序自动分词，生成 token 初稿；
2. 大模型补充 lemma、pos、meaningInContext；
3. 人工抽查重点句和高频词。

### 16.3 Token 与 Phrase 的关系

短语不能被简单拆成单词解释。

例如：

```text
eat his words
```

不能解释成：

```text
eat = 吃
words = 话
```

而应标注为：

```text
phrase meaning = 收回前言，承认自己说错了
```

Token 层应允许：

- token 属于某个 phrase；
- phrase 覆盖多个 token；
- 点击单词时，如果它属于短语，优先显示短语文中义。

## 22. 风险和应对

### 22.1 数据质量风险

风险：

- 结构拆分不一致；
- role 标签混乱；
- target_id 指错；
- 重点词没有绑定正确 chunk。

应对：

- 建立 role 白名单；
- 建立 JSON 校验脚本；
- 后台管理中提示数据错误；
- 每篇文章导入前先校验。

### 22.2 高亮失败风险

风险：

- chunk 英文无法在原句中匹配。

应对：

- 高亮功能降级；
- 匹配失败时仍展示结构表；
- 开发环境输出 warning；
- 后续通过导入脚本自动生成 start/end。

### 22.3 功能膨胀风险

风险：

- 一开始就想做登录、数据库、后台、AI，导致核心体验迟迟出不来。

应对：

- 第一阶段只做文章阅读、结构解析、基础检测；
- 其他功能只预留架构，不立即实现；
- 每个阶段都有明确验收标准。

### 22.4 AI 批改不稳定风险

风险：

- AI 评分不一致；
- AI 过度关注中文通顺，不关注结构；
- 成本较高。

应对：

- 先做客观题；
- 后做结构化主观题；
- 给 AI 明确评分 rubrics；
- 保存 AI 批改历史用于复查。
