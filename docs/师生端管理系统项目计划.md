# 师生端管理系统项目计划

本文档用于规划当前“考研长难句训练系统”从静态学习工具升级为可真实部署的教学管理系统。

当前项目已经具备学生学习页面、文章解析、练习入口和后台预留页。下一阶段的重点不是推翻现有结构，而是在现有 Next.js 项目上增加账号体系、角色权限、老师端、学生端、管理员端、班级、考试、互评批改和成绩归档。

## 1. 项目目标

系统最终应支持三类用户：

- 管理员：创建老师账号，维护系统词库，管理全局数据。
- 老师：管理班级、分配学生、创建考试、设置批改关系、查看成绩。
- 学生：使用现有学习页面，参加考试，提交答卷，批改被分配的答卷，查看成绩。

第一阶段目标是跑通教学闭环：

```text
管理员创建老师
  -> 老师获得邀请码
  -> 学生注册并输入老师邀请码
  -> 老师把学生分入班级
  -> 老师从词库创建单词释义默写考试
  -> 学生提交答卷
  -> 系统按既定关系分配批改任务
  -> 学生逐题批改
  -> 系统自动计算成绩
  -> 老师查看班级成绩和答卷详情
```

## 2. 已确认需求

### 2.1 用户与注册

- 系统可能公开部署。
- 需要管理员端。
- 老师账号由管理员创建。
- 学生通过老师邀请码注册。
- 学生使用真实姓名。
- 学生需要学号。
- 学生使用“学号或用户名 + 密码”登录。
- 学号建议全局唯一，减少公开部署后的重名和跨老师冲突问题。
- 学生选错老师后，由老师或管理员调整。
- 学生未分班前仍可使用学习功能。

### 2.2 老师、学生和班级

- 学生注册时先绑定老师。
- 班级由老师手动创建。
- 老师手动把学生一个一个分入班级。
- 一个老师可以管理多个班级。
- 一个班大约 15 个学生。
- 一个学生不能同时属于多个班级。
- 班级内学生按学号排序。

### 2.3 考试

- 学生端现有学习界面保留，增加考试入口。
- 老师端是全新的工作台界面。
- 考试类型先做“给定英文单词，填写中文释义”。
- 不考英文拼写。
- 题目来源支持：
  - 老师从现有词库手动勾选。
  - 系统自动生成。
- 自动生成应逐步支持：
  - 按单元生成。
  - 按真题年份和 Text 生成。
  - 按重点词生成。
  - 按最近学习内容生成。
  - 按错题或薄弱词生成。
  - 随机抽取指定数量。
- 考试开始时间、截止时间、答题限时、批改截止时间都由老师设置。
- 学生可以重复提交。
- 系统以最后一次提交为准。

### 2.4 批改与成绩

- 一个学生答卷由一个同学批改。
- 批改关系由老师设置。
- 支持“一键顺序分配 + 手动调整”。
- 允许学生批改自己的答卷。
- 学生提交后，自动流转给既定批改人。
- 如果批改人不批改，老师可以重新分配，或者最后让考生自己批改。
- 批改方式为逐题判断对错。
- 系统自动计算总分。
- 学生批改后的成绩直接生效。
- 老师不能直接改分。
- 老师可以查看每个学生的原始答案和批改详情。
- 学生可以看到自己的成绩。
- 被批改学生可以看到批改人。
- 成绩需要进入长期档案。

## 3. 产品边界

### 3.1 第一版必须做

- 管理员登录。
- 管理员创建老师账号。
- 老师邀请码。
- 学生注册和登录。
- 角色区分。
- 老师创建班级。
- 老师分配学生到班级。
- 学生端增加考试入口。
- 老师从词库创建考试。
- 老师设置考试时间和批改时间。
- 老师一键生成批改关系并可手动调整。
- 学生答题和重复提交。
- 学生批改被分配答卷。
- 系统自动算分。
- 老师查看班级成绩表。
- 老师查看单个学生答卷和批改详情。
- 学生查看个人成绩。

### 3.2 第一版暂缓

- 手机短信验证码。
- 微信登录。
- 支付系统。
- 家长端。
- 复杂 AI 批改。
- 老师改分。
- 多老师共同管理一个班。
- 一个学生加入多个班。
- 大规模批量导入学生。
- 复杂数据大屏。

### 3.3 建议保留但不做重

- 管理员重置密码。
- 老师禁用学生账号。
- 老师重新分配批改。
- 考试重开。
- 成绩导出 CSV。
- 操作日志。

## 4. 推荐技术路线

### 4.1 当前项目状态

当前项目技术栈：

```text
Next.js
TypeScript
React
zod
本地 JSON 数据
Prisma 目录预留
静态导出 output: "export"
```

现有功能大致为：

```text
/
  学生学习入口和文章选择

/articles/[articleId]
  长难句文章学习页

/practice/[articleId]
  精分析检测或练习页

/admin
  后台预留页
```

### 4.2 数据库选择

推荐使用 PostgreSQL。

原因：

- 用户、班级、考试、答卷、批改、成绩都是强关系数据。
- 后续公开部署时，比 SQLite 更适合并发和备份。
- Prisma 对 PostgreSQL 支持成熟。
- 可部署到 Supabase、Neon、Railway、Render、Vercel Postgres、自建服务器或云数据库。

不建议正式部署继续依赖纯 JSON 或 SQLite。JSON 可以继续作为系统词库的导入来源，但不应作为正式用户数据存储。

### 4.3 认证方案

推荐使用 Auth.js 或自建 credentials 登录。

第一版可以先做：

```text
用户名/学号 + 密码
```

不急着做：

```text
手机号验证码
邮箱验证码
第三方 OAuth
微信登录
```

公开部署后建议补充：

- 密码加密存储。
- 登录失败次数限制。
- 管理员重置密码。
- session 过期策略。
- 关键操作日志。

### 4.4 部署变化

当前项目使用：

```js
output: "export"
```

这适合纯静态站点。

接入数据库和登录后，需要改为动态部署：

```text
删除 output: "export"
使用 Next.js Route Handler / Server Action / Server Component
部署到支持服务端运行的平台
```

推荐部署路径：

```text
方案 A：Vercel + PostgreSQL 托管服务
方案 B：Render/Railway + PostgreSQL
方案 C：自建服务器 + PostgreSQL + Node.js 进程管理
```

## 6. 用户状态流转

### 6.1 老师账号

```text
管理员创建
  -> 老师账号可用
  -> 系统生成邀请码
  -> 老师登录
  -> 创建班级
  -> 管理学生和考试
```

老师状态：

```text
ACTIVE      正常
DISABLED    禁用
```

### 6.2 学生账号

```text
学生注册
  -> 输入老师邀请码
  -> 绑定老师
  -> 进入待分班
  -> 可使用学习功能
  -> 老师分班
  -> 可参加班级考试
```

学生状态：

```text
PENDING_CLASS     已绑定老师，待分班
ACTIVE            已分班，正常
TRANSFER_PENDING  转老师或转班处理中
DISABLED          禁用
```

### 6.3 考试状态

```text
DRAFT
  草稿，学生不可见

PUBLISHED
  已发布，学生可见

OPEN
  到达开始时间，可答题

SUBMISSION_CLOSED
  答题截止，不能再提交

GRADING_OPEN
  批改开放

GRADING_CLOSED
  批改截止

FINISHED
  成绩归档
```

第一版可以用较少状态：

```text
DRAFT
PUBLISHED
CLOSED
FINISHED
```

具体是否可答题和可批改，由时间字段判断。

- `unitCode` 用于后续按单元自动生成。

## 11. Prisma 草案

下面不是最终代码，但可以作为 `prisma/schema.prisma` 的第一版方向。

```prisma
enum UserRole {
  ADMIN
  TEACHER
  STUDENT
}

enum UserStatus {
  ACTIVE
  DISABLED
}

enum StudentStatus {
  PENDING_CLASS
  ACTIVE
  TRANSFER_PENDING
  DISABLED
}

enum ExamStatus {
  DRAFT
  PUBLISHED
  CLOSED
  FINISHED
}

enum SubmissionStatus {
  DRAFT
  SUBMITTED
}

enum GradingAssignmentStatus {
  WAITING_SUBMISSION
  READY
  GRADED
  EXPIRED
  REASSIGNED
}

model User {
  id           String     @id @default(cuid())
  username     String     @unique
  passwordHash String
  role         UserRole
  status       UserStatus @default(ACTIVE)
  createdAt    DateTime   @default(now())
  updatedAt    DateTime   @updatedAt

  adminProfile   AdminProfile?
  teacherProfile TeacherProfile?
  studentProfile StudentProfile?
}

model AdminProfile {
  id          String @id @default(cuid())
  userId      String @unique
  displayName String
  user        User   @relation(fields: [userId], references: [id])
}

model TeacherProfile {
  id             String   @id @default(cuid())
  userId         String   @unique
  realName       String
  invitationCode String   @unique
  createdAt      DateTime @default(now())

  user     User             @relation(fields: [userId], references: [id])
  classes  Class[]
  students StudentProfile[]
  exams    Exam[]
}

model StudentProfile {
  id        String        @id @default(cuid())
  userId    String        @unique
  realName  String
  studentNo String        @unique
  teacherId String
  classId   String?
  status    StudentStatus @default(PENDING_CLASS)
  createdAt DateTime      @default(now())
  updatedAt DateTime      @updatedAt

  user    User           @relation(fields: [userId], references: [id])
  teacher TeacherProfile @relation(fields: [teacherId], references: [id])
  class   Class?         @relation(fields: [classId], references: [id])
}

model Class {
  id          String   @id @default(cuid())
  teacherId   String
  name        String
  description String?
  status      String   @default("ACTIVE")
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  teacher  TeacherProfile  @relation(fields: [teacherId], references: [id])
  students StudentProfile[]
  exams    Exam[]

  @@unique([teacherId, name])
}

model VocabularyItem {
  id               String   @id @default(cuid())
  word             String
  meaning          String
  pos              String?
  sourceType       String
  sourceArticleId  String?
  sourceSentenceId String?
  sourceText       String?
  unitCode         String?
  examYear         Int?
  examType         String?
  textId           String?
  importanceLevel  String   @default("MEDIUM")
  createdAt        DateTime @default(now())
  updatedAt        DateTime @updatedAt

  examQuestions ExamQuestion[]
}

model Exam {
  id               String     @id @default(cuid())
  teacherId        String
  classId          String
  title            String
  description      String?
  status           ExamStatus @default(DRAFT)
  startAt          DateTime?
  submitDeadline   DateTime?
  gradingDeadline  DateTime?
  timeLimitMinutes Int?
  allowResubmit    Boolean    @default(true)
  scorePerQuestion Int        @default(1)
  createdAt        DateTime   @default(now())
  updatedAt        DateTime   @updatedAt
  publishedAt      DateTime?

  teacher     TeacherProfile      @relation(fields: [teacherId], references: [id])
  class       Class               @relation(fields: [classId], references: [id])
  questions   ExamQuestion[]
  submissions ExamSubmission[]
  assignments GradingAssignment[]
}

model ExamQuestion {
  id               String @id @default(cuid())
  examId           String
  vocabularyItemId String
  promptWord       String
  standardMeaning  String
  orderIndex       Int
  score            Int

  exam           Exam           @relation(fields: [examId], references: [id])
  vocabularyItem VocabularyItem @relation(fields: [vocabularyItemId], references: [id])
  answers        ExamAnswer[]
  gradingItems   GradingItem[]

  @@unique([examId, orderIndex])
}
```

后续还需要继续补 `ExamSubmission`、`ExamAnswer`、`GradingAssignment`、`GradingResult`、`GradingItem`、`AuditLog`。建议实际编码时分两次迁移，先完成用户/班级/词库，再完成考试/批改/成绩。

## 15. 批改分配算法

### 15.1 一键顺序分配

班级学生按学号排序：

```text
S1, S2, S3, S4, S5
```

默认顺序分配：

```text
S1 的答卷 -> S2 批改
S2 的答卷 -> S3 批改
S3 的答卷 -> S4 批改
S4 的答卷 -> S5 批改
S5 的答卷 -> S1 批改
```

允许自批时，也可以选择：

```text
S1 的答卷 -> S1 批改
S2 的答卷 -> S2 批改
```

但建议默认采用错位顺序分配，老师可以手动改成自批。

### 15.2 自动流转

老师发布考试时先保存批改关系。

学生提交后：

```text
找到 examId + submitterStudentId 的 GradingAssignment
如果存在 graderStudentId，则状态变为 READY
批改人学生端出现待批改任务
```

### 15.3 异常情况

```text
未提交：
  批改任务保持 WAITING_SUBMISSION

提交后批改人未批改：
  老师可重新分配
  或设置为提交者自批

学生重复提交：
  批改任务关联最新有效 submission
  如果旧批改已完成，需决定是否作废
```

建议第一版规则：

```text
在批改前重复提交：
  批改最新提交

已批改后再次提交：
  如果仍在答题截止前，旧成绩作废，批改状态回到 READY
  如果已经过答题截止，不允许再提交
```

## 16. 成绩规则

### 16.1 单题判分

每题由批改人选择：

```text
正确
错误
```

第一版不做半分。

### 16.2 总分计算

```text
总分 = 所有正确题目的 score 之和
满分 = 所有题目的 score 之和
百分比 = 总分 / 满分 * 100
```

### 16.3 成绩生效

学生提交批改后：

```text
创建 GradingResult
创建 GradingItem
计算 totalScore
写入或更新 ExamScore
写入 StudentScoreArchive
```

老师不能直接修改成绩。

### 16.4 推荐补充机制

虽然老师不能改分，但建议保留：

```text
重新分配批改
作废异常批改
重开该学生批改任务
记录操作日志
```

这不是“老师改分”，而是处理误批或未批改的必要管理动作。

## 17. 数据迁移计划

当前系统词汇主要来自文章 JSON 的 `key_words`。迁移时可按以下顺序。

### 17.1 阶段一：保留 JSON 学习数据

```text
文章、句子、chunks、重点词继续从 data/articles 读取
用户、班级、考试、答卷进入 PostgreSQL
```

优点：

- 改动小。
- 不影响现有学习页。
- 可先完成教学管理闭环。

### 17.2 阶段二：导入词库

写脚本从文章 JSON 提取：

```text
key_words[].word
key_words[].meaning
key_words[].pos
sentence.year
sentence.exam_type
sentence.text_id
sentence.sentence_id
sentence.original
```

生成 `VocabularyItem`。

### 17.3 阶段三：文章数据入库

后续再把：

```text
Article
Sentence
Chunk
KeyWord
KeyPhrase
```

全部迁移到 PostgreSQL。

页面通过 repository 抽象读取，避免页面大改。

## 19. 实施阶段

### 阶段 0：技术准备

目标：把项目从静态学习工具过渡到可接数据库的结构。

任务：

- 移除或准备移除 `output: "export"`。
- 安装 Prisma。
- 配置 PostgreSQL。
- 配置 `.env`。
- 建立数据库连接工具。
- 保留现有 JSON 学习功能不变。

验收标准：

- 本地开发可连接 PostgreSQL。
- 现有学习页仍可打开。
- 可以运行一次空迁移或基础迁移。

### 阶段 1：账号和权限

目标：实现三类用户登录。

任务：

- 建立 User、AdminProfile、TeacherProfile、StudentProfile。
- 实现密码哈希。
- 实现登录。
- 实现角色鉴权。
- 实现管理员种子账号。
- 管理员可创建老师账号。
- 创建老师时生成邀请码。
- 学生注册时输入邀请码。

验收标准：

- 管理员、老师、学生登录后进入不同页面。
- 学生注册后绑定正确老师。
- 学生未分班时仍能进入学习首页。
- 非法角色不能访问对应页面。

### 阶段 2：老师端班级管理

目标：老师可以管理班级和学生。

任务：

- 老师创建班级。
- 老师查看待分班学生。
- 老师把学生分入班级。
- 老师查看班级学生列表。
- 学生按学号排序。
- 学生只能属于一个班级。

验收标准：

- 一个老师能创建多个班级。
- 学生注册后出现在该老师待分班列表。
- 分班后出现在班级详情页。
- 学生端能看到自己的老师和班级状态。

### 阶段 3：系统词库

目标：考试可以从词库出题。

任务：

- 建立 VocabularyItem。
- 从现有文章 JSON 导入重点词。
- 管理员可查看词库。
- 老师创建考试时可筛选词库。
- 支持按年份、英语一/英语二、Text、单元、关键词筛选。

验收标准：

- 词库中能看到当前文章重点词。
- 老师可以勾选词生成考试题。
- 已发布考试保存题目快照。

### 阶段 4：老师创建考试

目标：老师能发布单词释义默写考试。

任务：

- 创建 Exam、ExamQuestion。
- 老师选择班级。
- 老师设置考试时间、答题截止、批改截止、限时。
- 老师从词库勾选单词。
- 支持草稿和发布。
- 学生端考试列表显示对应考试。

验收标准：

- 老师能创建并发布考试。
- 只有该班学生能看到考试。
- 未分班学生看不到班级考试。

### 阶段 5：学生答题

目标：学生可以完成考试。

任务：

- 学生考试列表。
- 学生答题页。
- 保存草稿。
- 提交答卷。
- 支持重复提交。
- 以最后一次提交为准。
- 到截止时间后禁止提交。

验收标准：

- 学生能填写每个单词的中文释义。
- 提交后老师端能看到提交状态。
- 重复提交后最新答案生效。
- 截止后不能再提交。

### 阶段 6：批改分配

目标：老师设置批改关系，学生提交后自动流转。

任务：

- 手动调整批改人。

验收标准：

## 22. 风险和应对

### 22.1 公开部署带来的账号风险

风险：

- 外部用户乱注册。
- 老师身份被冒用。

应对：

- 老师只能由管理员创建。
- 学生必须输入老师邀请码。
- 管理员可禁用账号。

### 22.2 学号重复风险

风险：

- 公开部署后不同老师可能有相同学号。

建议：

- 第一版使用全局唯一学号。
- 如果未来多机构使用，可改为“机构内唯一”。

### 22.3 学生乱批风险

风险：

- 学生随意判断对错，成绩直接生效。

应对：

- 老师可查看批改详情。
- 保留重新分配批改。
- 保留作废异常批改。
- 记录批改人。

### 22.4 已批改后重复提交风险

风险：

- 学生提交新答案后，旧批改结果是否还有效不清晰。

建议规则：

- 答题截止前允许重复提交。
- 如果新提交发生在批改前，批改最新版本。
- 如果新提交发生在批改后，旧成绩作废，重新进入待批改。
- 答题截止后禁止提交。

### 22.5 当前静态导出限制

风险：

- `output: "export"` 无法承载数据库登录和动态 API。

应对：

- 接数据库阶段切换为动态 Next.js 部署。
- 更新部署平台和 `netlify.toml`。

### 22.6 功能过多导致延期

风险：

- 管理员端、老师端、学生端、考试、批改、成绩同时做，容易失控。

应对：

- 按里程碑推进。
- 每个里程碑都有可演示结果。
- 第一版先做单词中文释义默写，不扩展其他题型。

## 24. 后续待确认问题

这些问题不影响启动第一版，但会影响后续扩展：

```text
1. 是否需要机构概念？
   如果未来多个学校或多个机构使用，需要 Organization 表。

2. 管理员是否只有一个？
   如果多个管理员，需要更细的管理员权限。

3. 学生是否需要转老师历史？
   如果需要，应保留 StudentTeacherHistory。

4. 词库是否需要同义答案？
   例如 monarchy 可以接受“君主制”和“君主制度”。

5. 批改是否允许备注？
   第一版可以不做，后续可让批改人写简短说明。

6. 成绩是否允许导出 Excel？
   第一版 CSV 即可，后续再做 xlsx。

7. 自动组卷是否要避开已考过单词？
   第二版建议加入。
```

## 25. 下一步执行建议

建议下一步不要直接开写所有功能，而是按以下顺序执行：

```text
1. 修改技术基座：接入 PostgreSQL 和 Prisma。
2. 建账号体系：User、角色、登录、管理员种子账号。
3. 建老师创建流程：管理员创建老师，生成邀请码。
4. 建学生注册流程：邀请码绑定老师，待分班。

9. 建批改分配：一键顺序分配和手动调整。

11. 建成绩表：老师端查看，学生端查看。
```

完成第 11 步后，这个系统就具备可真实使用的第一版教学闭环。
