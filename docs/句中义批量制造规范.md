# 句中义批量制造规范

本文档用于指导后续 7 篇文章的 `meaning_in_context` 数据批量制造。

目标不是把整篇文章所有词都做一遍，而是稳定产出一套适合学生训练的“句中义判断”题库。重点是：

- 让学生根据语境掌握相对准确的意思。
- 避免机械套词条、逐词硬译。
- 保持输出风格统一，不因文章不同而越来越粗糙。

## 1. 适用范围

适用对象：

- `英语一json/kaoyan_2015_english1_text2.json`
- `英语一json/kaoyan_2015_english1_text3.json`
- `英语一json/kaoyan_2015_english1_text4.json`
- `英语二json/kaoyan_2015_english2_text1.json`
- `英语二json/kaoyan_2015_english2_text2.json`
- `英语二json/kaoyan_2015_english2_text3.json`
- `英语二json/kaoyan_2015_english2_text4.json`

不再要求改动现有文章 JSON。

每篇文章输出一个独立文件，例如：

```text
data/practice/kaoyan_2015_english1_text2_meaning_in_context.json
```

## 2. 输出目标

每个文件都是一个 JSON 数组，数组里每一项都是一个句中义判断点。

示例字段：

```json
[
  {
    "check_id": "s_003_suggest_meaning_in_context",
    "source_sentence_id": "s_003",
    "target": "suggest",
    "target_type": "word",
    "prompt": "结合本句语境，suggest 最合适的意思是：",
    "correct_meaning": "表明/暗示",
    "acceptable_meanings": ["表明", "暗示", "说明"],
    "meaning_pool": ["建议", "表明/暗示", "推荐", "使想起"],
    "explanation": "suggest 后接 that 宾语从句，说明本句中它的句中义是“表明/暗示”。",
    "priority": 1
  }
]
```

## 3. 总原则

### 3.1 只做第一优先级

只选那些**看起来简单，但放进句子里容易理解偏、翻译硬、影响整句理解**的词或短语。

包括但不限于：

- 小词：`as`、`while`、`with`、`following`、`excepted`
- 常见词但句中义重要：`suggest`、`case`、`mean`、`claim`
- 固定短语：`eat his words`、`the writing is on the wall`、`have most to fear`
- 抽象语境中的常见词：`service`、`survive`、`public figure`

不选这类：

- 区分度很低的词
- 只差一点点语气、学生答错也不太影响理解的词
- 只是“好像有差异”，但不值得单独设题的词

### 3.2 数量要够，但不要全做

每篇文章建议：

- **最少 18 个**
- **常规 20-25 个**
- **最多 30 个**

不是每个句子都要有点，也不是每个重点词都必须上。

### 3.3 不要复杂化

不要再拆成很多层：

- 不要 `trap_reason`
- 不要过细的错误标签
- 不要一个词下面再分太多子类

这一套只保留一句话的检查目标：

> 结合语境，判断这个词或短语在本句中的相对准确意思。

## 4. 字段约束

统一字段如下：

```json
{
  "check_id": "",
  "source_sentence_id": "",
  "target": "",
  "target_type": "word | phrase | function_word | structure",
  "prompt": "",
  "correct_meaning": "",
  "acceptable_meanings": [],
  "meaning_pool": [],
  "explanation": "",
  "priority": 1
}
```

### 4.1 `check_id`

命名建议：

```text
句子ID_目标_含义_in_context
```

例子：

- `s_003_suggest_meaning_in_context`
- `s_016_while_meaning_in_context`

要求：

- 全局唯一
- 尽量能看出题目是什么

### 4.2 `source_sentence_id`

必须能在原文章 JSON 中找到对应句子。

### 4.3 `target_type`

只保留 4 类：

- `word`
- `phrase`
- `function_word`
- `structure`

不要再继续细分。

### 4.4 `prompt`

统一口径：

```text
结合本句语境，XXX 最合适的意思是：
```

如果是结构，就写：

```text
结合本句语境，XXX 最合适的理解是：
```

### 4.5 `correct_meaning`

本句最推荐的答案。

要求：

- 简短
- 自然
- 符合语境
- 不要太口语化

### 4.6 `acceptable_meanings`

这是**可接受答案**，表示学生这样写也算对。

比如：

- `表明`
- `暗示`
- `说明`

### 4.7 `meaning_pool`

这是**选项来源池**，用于后面出选择题。

要求：

- 至少 4 个
- 里面要有正确项
- 也要有几个看起来像、但不合适的义项
- 不要太离谱

### 4.8 `explanation`

只写一两句，说明为什么本句要取这个意思。

不要长篇大论。

### 4.9 `priority`

统一写 `1`。

## 5. 选词标准

优先挑这几类：

### 5.1 套常见义会错的词

例如：

- `suggest`
- `case`
- `arguments`
- `mean`

### 5.2 看起来简单，但句中义很关键的小词

例如：

- `as`
- `while`
- `with`
- `following`
- `excepted`

### 5.3 固定搭配或习语

例如：

- `eat his words`
- `the writing is on the wall`
- `have most to fear`
- `rise above politics`

### 5.4 抽象语境中的常见词

例如：

- `service`
- `survive`
- `public figure`
- `downside`

## 6. 不要选的词

以下情况尽量不选：

- 只是“词典里义项很多”，但本句没有明显区分度
- 学生就算答成别的也不太影响文章理解
- 只是普通词，不足以作为训练点
- 解释一写就很长，说明它不适合做这一层的检查

举例：

- `embody`
- `apparent`
- `explains`
- `insisted`

这些可以作为普通重点词出现，但不优先做 `meaning_in_context`。

## 7. 文章内数量控制

每篇文章不必均匀分布。

建议：

- 重要句子可以 2-3 个点
- 普通句子可以 0-1 个点
- 个别句子没有点也正常

不要为了均匀，把每句都硬塞一个点。

## 8. 排序原则

建议按句子顺序输出，不要乱跳。

原因：

- 方便核对原文
- 方便后续滚动复习
- 方便老师批量检查

## 9. 质量底线

每篇文章完成后，至少检查：

1. `source_sentence_id` 是否真实存在。
2. `correct_meaning` 是否能被 `acceptable_meanings` 覆盖。
3. `meaning_pool` 是否包含正确项。
4. `prompt` 是否统一、简洁。
5. `explanation` 是否一句话讲清楚。
6. 数量是否在 18-30 之间。
7. 是否没有把不值得考的词也塞进去。

## 10. 批量生产流程

建议按下面顺序做：

1. 先看原文章 JSON。
2. 标出 18-30 个最有价值的句中义点。
3. 按当前规范生成一个独立 `meaning_in_context` 文件。
4. 跑校验脚本。
5. 如果数量太少，再补；如果太散，再删。
6. 一篇做完再做下一篇。

不要同时开 7 篇一起写，不然风格会漂。

## 11. 给另一个聊天的执行说明

可以直接复制下面这段给另一个聊天：

```text
请按《句中义批量制造规范》为指定文章生成独立的 meaning_in_context JSON。

要求：
1. 只选第一优先级的句中义点，不要凑数。
2. 每篇文章 18-30 个点，推荐 20-25 个。
3. 不要 trap_reason。
4. 字段固定为：check_id, source_sentence_id, target, target_type, prompt, correct_meaning, acceptable_meanings, meaning_pool, explanation, priority。
5. prompt 统一写成“结合本句语境，XXX 最合适的意思是：”。
6. correct_meaning 要简短、准确、自然。
7. acceptable_meanings 是学生主观回答可接受的说法。
8. meaning_pool 是后续出选择题的候选义项，必须包含正确项。
9. explanation 只写一句，说明为什么在本句中要取这个意思。
10. 不要修改原文章 JSON。
11. 输出文件名必须是 data/practice/{source_basename}_meaning_in_context.json。
12. 做完后自己检查数量、重复、和原文句号是否对得上。
```
