# 长难句 JSON 数据规范 v1.3

本文档用于固定考研英语长难句分析项目的 JSON 标注规范。它沉淀的是前期讨论后的当前版本，目标是让后续继续生成 2016、2017 等年份数据时，格式稳定、字段克制、教学价值明确。

## 1. 设计目标

本项目不是普通翻译工具，而是“考研英语长难句结构训练系统”。

JSON 数据应支持：

- 按句子结构展示结构直译。
- 展示主语、谓语、宾语、定语、状语、补语、从句等结构块。
- 表达定语、状语、补语、从句等成分具体作用于哪个词或短语。
- 点击重点词、重点短语时显示文中义。
- 标记不容易看懂或容易误判的特殊句型。
- 后续自动生成语法、词汇、结构识别类练习。

## 2. 设计取舍

### 2.1 不做过大的 JSON

第一版不要把所有能力都塞进一个巨大 JSON。过大的结构会导致大模型生成时出现字段遗漏、层级混乱、id 对不上、校对成本过高等问题。

当前版本只保留最核心的句子解析字段：

- `chunks`
- `translation_literal`
- `key_words`
- `key_phrases`
- `sentence_patterns`

练习题暂时不放进句子解析 JSON，后续可以单独存储或由程序自动生成。

### 2.2 不做全文逐词 token

不要求每个单词都生成 token。原因是：

- 数据量会膨胀。
- `the`、`of`、`and` 等功能词没有必要逐个解释。
- 很多表达必须按短语理解，例如 `eat his words`、`stand down`、`the writing is on the wall`。

当前版本只标注重点词和重点短语。

### 2.3 结构块优先于单词对齐

中英对应以结构块为单位，而不是单词对单词。

原因是英语和中文语序经常不同，单词级对齐容易混乱。结构块对齐更适合长难句训练，例如：

- 主语对应主语
- 谓语对应谓语
- 定语对应被修饰名词
- 从句对应从句
- 补语对应被补充说明的对象

### 2.4 只保留结构直译

当前版本不再保留 `translation_smooth`。

原因是本项目训练的是长难句结构，不是文学化翻译。`translation_literal` 应尽量按照英文结构顺序翻译，保留主谓宾定状补关系，同时保证中文基本通顺。

## 3. 顶层结构

每个 JSON 文件是一个句子数组。

示例：

```json
[
  {
    "sentence_id": "s_001",
    "year": 2015,
    "exam_type": "英语一",
    "text_id": "Text 1",
    "original": "",
    "difficulty": 1,
    "translation_literal": "",
    "chunks": [],
    "sentence_patterns": [],
    "key_words": [],
    "key_phrases": []
  }
]
```

## 4. 句子字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `sentence_id` | string | 是 | 句子编号，例如 `s_001` |
| `year` | number | 是 | 真题年份，例如 `2015` |
| `exam_type` | string | 是 | `英语一` 或 `英语二` |
| `text_id` | string | 是 | 文章编号，例如 `Text 1` |
| `original` | string | 是 | 英文原句 |
| `difficulty` | number | 是 | 难度，建议 1-5 |
| `translation_literal` | string | 是 | 结构直译 |
| `chunks` | array | 是 | 句子结构块 |
| `sentence_patterns` | array | 是 | 特殊句型标记；没有则为空数组 |
| `key_words` | array | 是 | 重点词；没有则为空数组 |
| `key_phrases` | array | 是 | 重点短语；没有则为空数组 |

## 5. chunks 结构块规范

### 5.1 字段结构

```json
{
  "chunk_id": "c_001_001",
  "role": "主语",
  "english": "",
  "chinese": "",
  "parent_id": null,
  "target_id": null,
  "relation": null
}
```

### 5.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `chunk_id` | string | 是 | 结构块编号，同一句内唯一 |
| `role` | string | 是 | 中文语法成分标签 |
| `english` | string | 是 | 英文结构块原文 |
| `chinese` | string | 是 | 对应中文直译 |
| `parent_id` | string/null | 是 | 所属的大结构块 |
| `target_id` | string/null | 是 | 作用、修饰、补充或对应的目标结构块 |
| `relation` | string/null | 是 | 与目标之间的关系说明 |

### 5.3 parent_id 和 target_id

`parent_id` 表示“属于谁”。

例如主语整体是：

```text
embarrassing scandals and the popularity of the republican left in the recent Euro-elections
```

其中 `embarrassing` 属于这个主语整体，因此它的 `parent_id` 指向主语 chunk。

`target_id` 表示“作用于谁”。

例如：

- 定语修饰名词。
- 状语修饰谓语、形容词或整个主句。
- 宾语补足语补充说明宾语。
- 定语从句修饰先行词。
- 主语从句对应形式主语。
- 真正宾语对应形式宾语。

`target_id` 比 `modifies_id` 更合适，因为不是所有关系都叫“修饰”。补语、同位语、形式主语对应关系都更适合用通用的 `target_id`。

### 5.4 role 标签

当前允许使用的主要 role：

```text

状语从句
从句主语
从句谓语
从句宾语

定语从句谓语
定语从句宾语
定语从句表语
同位语从句主语
同位语从句谓语
同位语从句宾语

```

后续可以扩展，但不要随意创造过细标签。标签越多，大模型越容易不稳定。

### 5.5 拆分粒度

应该拆：

- 主干成分：主语、谓语、宾语、表语、补语。
- 重要修饰成分：定语、状语、同位语、插入语。
- 影响理解的从句。
- 从句内部的主语、谓语、宾语、表语，尤其是长难句。
- 长介词短语。
- 影响理解的非谓语结构。
- 重要并列结构。

不必拆：

- 每个普通功能词。
- 对理解没有帮助的极细碎词块。
- 只为了增加数量而做的“重点词定位” chunk。

## 6. key_words 重点词规范

### 6.1 字段结构

```json
{
  "word": "",
  "pos": "v.",
  "meaning": "",
  "chunk_id": ""
}
```

### 6.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `word` | string | 是 | 原文中的单词或必要时的词形 |
| `pos` | string | 是 | 词性缩写 |
| `meaning` | string | 是 | 文中义 |
| `chunk_id` | string | 是 | 该词所在或最相关的 chunk |

### 6.3 词性列表

重点词词性使用英文缩写：

```text
n.
v.
adj.
adv.
pron.
prep.
conj.
det.
modal v.
num.
abbr.
```

说明：

- 名词用 `n.`
- 动词用 `v.`
- 形容词用 `adj.`
- 副词用 `adv.`
- 代词用 `pron.`
- 介词用 `prep.`
- 连词用 `conj.`
- 限定词用 `det.`
- 情态动词用 `modal v.`
- 数词用 `num.`
- 缩写用 `abbr.`

如果一个词在文中具有短语化意义，优先放入 `key_phrases`，不要硬拆成普通单词解释。

## 7. key_phrases 重点短语规范

### 7.1 字段结构

```json
{
  "phrase": "",
  "phrase_type": "v. phr.",
  "meaning": "",
  "chunk_id": ""
}
```

### 7.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `phrase` | string | 是 | 原文短语 |
| `phrase_type` | string | 是 | 短语类型 |
| `meaning` | string | 是 | 文中义 |
| `chunk_id` | string | 是 | 该短语所在或最相关的 chunk |

### 7.3 短语类型建议

```text
n. phr.
v. phr.
adj. phr.
adv. phr.
prep. phr.
idiom
collocation
sentence pattern
```

说明：

- 名词短语用 `n. phr.`
- 动词短语用 `v. phr.`
- 形容词短语用 `adj. phr.`
- 副词短语用 `adv. phr.`
- 介词短语用 `prep. phr.`
- 习语用 `idiom`
- 固定搭配用 `collocation`
- 句型表达可用 `sentence pattern`

例如：

```json
{
  "phrase": "eat his words",
  "phrase_type": "idiom",
  "meaning": "收回前言，承认自己说错了",
  "chunk_id": "c_002_010"
}
```

## 8. sentence_patterns 特殊句型规范

### 8.1 字段结构

```json
{
  "pattern_name": "形式主语",
  "pattern_form": "it is + adj. + that 从句",
  "explanation": "it 是形式主语，真正主语是后面的 that 从句。"
}
```

### 8.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `pattern_name` | string | 是 | 句型名称 |
| `pattern_form` | string | 是 | 抽象句型形式 |
| `explanation` | string | 是 | 面向考生的解释 |

不使用 `pattern_english` 和 `pattern_chinese`，因为原句和直译已经存在，不需要重复存例子。

不使用 `chunk_ids`，因为当前阶段特殊句型只需要说明该句存在什么结构，不需要做复杂联动。

### 8.3 需要标记的句型

只标记不那么好理解、容易看错、对长难句分析有价值的结构：

```text
强调句
形式主语
形式宾语
倒装句
让步倒装
省略结构
插入语结构
as 引导的特殊结构
as...as...
分裂结构 / 间隔结构
同位语解释结构
长并列结构
```

### 8.4 不需要标记的普通句型

以下结构暂时不纳入 `sentence_patterns`：

```text
more...than...
not so much A as B
not...but...
so...that...
such...that...
enough to do
too...to do
rather than
instead of
```

这些结构可以作为重点短语或普通语法说明处理，但不作为当前版本的特殊句型。

## 10. 质量检查规则

每次生成或修改 JSON 后，应检查：

- JSON 可以正常解析。
- 每个句子都有 `year`、`exam_type`、`text_id`。
- 每个句子都有 `sentence_patterns`，没有特殊句型时为空数组。
- 每个 `key_words` 都有 `pos`。
- 每个 `key_phrases` 都有 `phrase_type`。
- 每个 `chunk_id` 在同一句内唯一。
- 非空 `parent_id` 必须指向同一句内存在的 `chunk_id`。
- 非空 `target_id` 必须指向同一句内存在的 `chunk_id`。
- `key_words[].chunk_id` 必须指向同一句内存在的 `chunk_id`。
- `key_phrases[].chunk_id` 必须指向同一句内存在的 `chunk_id`。
- 不应出现只为定位重点词而添加的无教学价值 chunk。

## 11. 后续扩展方向

后续如果要继续升级，可以考虑：

- 单独建立 `exercises.json` 存储题库。
- 增加人工精拆版，对每篇 5-8 个最长句做更细的从句内部结构。
- 将词性、短语类型、特殊句型写入代码枚举校验。
- 为每篇文章生成质量报告，统计结构密度、从句内部拆分数量、断裂引用等。
