# v1.3 数据生成质量门禁

本文档用于约束后续生成 2016、2017 等年份阅读 JSON 数据。它不是字段说明，而是生成前、生成中、生成后的质量控制流程。

结构字段以 [v1.3-data-spec.md](./v1.3-data-spec.md) 为准；本文档负责防止数据粗糙、重复拆分、从句内部不拆、关系挂错等问题。

## 1. 核心原则

生成数据时，目标不是“看起来字段很多”，而是让学生能看懂：

- 主干在哪里。
- 修饰成分修饰谁。
- 从句内部主谓宾是什么。
- 并列结构哪几部分并列。
- 特殊句型为什么容易误判。
- 重点词和短语在文中具体是什么意思。

凡是不能帮助学生理解句子结构的 chunk，都不应该为了增加数量而加入。

## 2. 生成流程

每篇文章按下面流程生成。

### 2.1 先切句

先把文章切成句子数组，确认：

- 没有漏句。
- 没有把一个完整句子切断。
- 引号、冒号、分号后的结构没有被误切。
- 每个句子按原文顺序编号为 `s_001`、`s_002`。

### 2.2 每句先找主干

每个句子必须先确定主干：

- 主语
- 谓语
- 宾语 / 表语 / 补语

不要一上来拆定语、状语、从句。主干不清楚时，后面的 `parent_id` 和 `target_id` 很容易乱。

### 2.3 再拆修饰和从句

主干确定后，再拆：

- 定语：必须写清楚修饰哪个名词或名词短语。
- 状语：必须写清楚修饰谓语、形容词还是整个分句。
- 补语：必须写清楚补充说明主语还是宾语。
- 同位语：必须写清楚解释说明哪个名词。
- 从句：必须写清楚充当什么成分，或者修饰哪个先行词。

### 2.4 长句必须精拆从句内部

只把一个长从句整体标成 `宾语从句`、`定语从句`、`同位语从句` 是不够的。

遇到以下情况，必须继续拆从句内部：

- 从句长度超过 8 个英文词。
- 从句中还有介词短语、非谓语、并列结构。
- 从句本身是理解难点。
- 从句中有容易误判的主谓宾。

从句内部至少应拆：

- `从句主语`
- `从句谓语`
- `从句宾语` / `从句表语`

如果是具体类型从句，也可以用：

- `定语从句主语`
- `定语从句谓语`
- `同位语从句主语`
- `同位语从句谓语`

但不要为了标签好看而混用。优先保持清楚。

## 3. 一票否决问题

生成后的 JSON 如果出现以下情况，应视为不合格，需要返工。

### 3.1 只拆整体从句，不拆内部

错误示例：

```json
{
  "role": "同位语从句",
  "english": "that basic mistakes in data analysis are contributing to the irreproducibility of many published research findings"
}
```

如果这个从句很长，必须继续拆：

- `basic mistakes in data analysis` 作从句主语
- `are contributing to` 作从句谓语
- `the irreproducibility...` 作从句宾语
- `in data analysis` 修饰 `mistakes`

### 3.2 同一个结构重复标注

不要把同一个英文片段以同一语法身份重复标注两次。

错误示例：

```text
主语 Readers
谓语 must have
宾语 confidence in the conclusions...
主语 Readers
谓语 must have
```

如果是为了拆宾语内部结构，应该拆 `confidence`、`in the conclusions...`、`published in our journal`，不要把主句主谓再重复一遍。

### 3.3 relation 空泛

以下 relation 不合格：

```text
主干或独立结构
修饰成分
补充说明
从句内部结构
```

合格 relation 必须具体：

```text
宾语从句内部主语
第一并列分句的谓语
修饰 conclusions
补充说明宾语 him 被迫做的动作
真正主语，对应形式主语 it
```

### 3.4 并列结构不说明归属

如果一个句子里有两个主语、两个谓语、两个宾语，必须说明它们属于哪个分句或哪个并列项。

例如：

```text
第一并列分句的主语
第二并列分句的主语
并列主语第一项
并列主语第二项
```

不能简单都写成 `主语`，否则前端展示时会让学生以为结构混乱。

### 3.5 从句内部挂载错误

从句内部的主语、谓语、宾语、表语必须通过 `parent_id` 挂到对应从句 chunk。

错误：

```text
从句主语 parent_id = 主句 chunk
```

正确：

```text
从句主语 parent_id = 对应的宾语从句 / 定语从句 / 同位语从句 chunk
```

### 3.6 key_words 和 key_phrases 绑定错误

重点词和重点短语的 `chunk_id` 必须指向它所在或最相关的结构块。

不要为了方便全部绑定到主句主语或句子第一个 chunk。

## 4. 每句最低质量要求

每个句子至少应满足：

- 有主干结构。
- `translation_literal` 能体现主干顺序。
- 每个非空 `parent_id` 和 `target_id` 都有效。
- 每个重点词有 `pos`。
- 每个重点短语有 `phrase_type`。
- `relation` 不使用空泛模板。

简单句可以只有 3-5 个 chunks。

复杂句通常应有 7-15 个 chunks。不要机械追求数量，但长句低于 6 个 chunks 时必须人工复查。

## 5. 长难句加强要求

每篇文章生成后，必须选出最复杂的 5-8 个句子做加强检查。

判断标准：

- 句子很长。
- 包含多个从句。
- 包含插入语、破折号、冒号、分号。
- 包含形式主语、形式宾语、强调句、倒装、省略。
- 包含长并列结构。
- 包含多层介词短语或非谓语结构。

这些句子必须做到：

- 主句主干清楚。
- 每个从句内部至少拆出主语和谓语。
- 长宾语、长主语、长表语内部要拆核心词和关键修饰。
- 并列结构要拆出并列项。
- `sentence_patterns` 必须标出容易误判的特殊结构。

## 6. sentence_patterns 标注要求

只标记 v1.3 规定的特殊句型：

```text
强调句
形式主语
形式宾语
倒装句
让步倒装
省略结构
插入语结构
as 引导的特殊结构
as...as...
分裂结构 / 间隔结构
同位语解释结构
长并列结构
```

不要把普通搭配硬塞进 `sentence_patterns`。

`explanation` 必须说明学生容易看错的点。例如：

```text
it 是形式主语，真正主语是后面的 that 从句。
```

不要写成：

```text
这是形式主语结构。
```

## 7. 生成用提示词模板

生成单篇文章 JSON 时，建议使用以下提示词。

```text
请按照 docs/v1.3-data-spec.md 和 docs/v1.3-generation-quality-gate.md 生成本文 JSON。

硬性要求：
1. 输出必须是 JSON 数组，不要输出解释文字。
2. 每个句子包含 sentence_id、year、exam_type、text_id、original、difficulty、translation_literal、chunks、sentence_patterns、key_words、key_phrases。
3. role 使用中文标签，尽量从 v1.3-data-spec.md 的 role 列表中选择。
4. key_words 必须包含 word、pos、meaning、chunk_id。
5. key_phrases 必须包含 phrase、phrase_type、meaning、chunk_id。
6. sentence_patterns 只标记 v1.3 允许的特殊句型，没有则为空数组。
7. 长从句不能只整体标注，必须拆内部主语、谓语、宾语或表语。
8. 并列结构必须写清楚第一项、第二项，或第一分句、第二分句。
9. 不要重复标注同一个英文片段；整体 chunk 可以有子 chunk，但子 chunk 必须通过 parent_id 挂到整体 chunk 下。
10. relation 必须具体说明结构关系，禁止使用“主干或独立结构”“修饰成分”“补充说明”这类空泛表述。
11. 不要生成练习题，不要生成 start_index/end_index。

文章信息：
year: 2016
exam_type: 英语一/英语二
text_id: Text X

原文：
...
```

## 8. 生成后自检清单

大模型生成后，必须逐篇检查：

- 是否漏句。
- 是否每句都有 `sentence_patterns`。
- 是否每个重点词都有 `pos`。
- 是否每个重点短语都有 `phrase_type`。
- 是否存在 `parent_id` / `target_id` 断裂。
- 是否存在空泛 relation。
- 是否存在明显重复的 chunk。
- 是否长句没有拆从句内部。
- 是否 key_words / key_phrases 绑定到无关 chunk。
- 是否特殊句型乱标或漏标。

项目中已经提供质量检查命令：

```bash
npm run validate:data
npm run quality:data
```

`validate:data` 检查 JSON 格式和 id 引用关系。

`quality:data` 检查教学质量风险，包括：

- `role` 是否在当前 2015 样板库确认过的白名单中，防止继续随意创造新标签。
- 长从句是否缺少内部主谓拆分。
- `parent_id` / `target_id` 存在时 `relation` 是否为空。
- 是否存在空泛 relation。
- 是否存在同一英文片段、同一 role 的重复 chunk。
- 重点词词性是否在白名单中。
- 重点短语类型是否在白名单中。
- 特殊句型是否在 v1.3 白名单中。

生成 2016 年数据时，每完成一篇 Text，都必须先运行这两个命令。`quality:data` 报出的内容需要人工判断；如果是长从句未拆、relation 为空、重复 chunk，应优先修正。

## 9. 建议质量指标

这些指标不是绝对标准，但可以辅助判断粗糙程度：

| 指标 | 建议 |
|---|---|
| 每句 chunks | 简单句 3-5，复杂句 7-15 |
| 长句 chunks | 不宜低于 8 |
| 从句内部拆分 | 长从句必须有 |
| 断裂引用 | 必须为 0 |
| 空泛 relation | 必须为 0 |
| 重点词 pos 缺失 | 必须为 0 |
| 重点短语 phrase_type 缺失 | 必须为 0 |
| 无意义重复 chunk | 必须为 0 |

不要只用平均 chunks 数判断质量。平均数可以参考，但不能代替人工检查。

## 10. 推荐工作方式

生成 2016 年数据时，建议不要一次性生成 8 篇后再检查。

推荐流程：

1. 先生成 `2016 英语一 Text 1`。
2. 立即做自动校验和人工抽查。
3. 修正提示词或规范问题。
4. 再批量生成剩余 Text。
5. 每生成一篇就跑校验。
6. 最后统一生成质量报告。

这样可以避免同一个错误在多个文件中批量复制。
